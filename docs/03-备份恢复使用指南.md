## 一、核心概念

### 1.1 xtrabackup 是什么

xtrabackup 是 Percona 开发的 MySQL 热备工具，比 mysqldump 更快，支持**增量备份**和**热备**（备份时 MySQL 不需要停机）。

### 1.2 LSN（Log Sequence Number）

LSN 是 MySQL 存储引擎（InnoDB）的内部版本号，类似于"时间戳"。每个数据页都有一个 LSN，记录页的最后修改位置。

**增量备份的原理：**
- 全量备份记录了**某一时刻**的 LSN，比如 0 → 19790477
- 增量备份只备份**从这个 LSN 之后**变化的数据页
- 增量备份记录新的 LSN，比如 19790477 → 19790866

查看备份的 LSN：
```bash
cat /data/backup/full_20260501_210000/xtrabackup_checkpoints
```

```
backup_type = full_backup
from_lsn = 0
to_lsn = 19790477
last_lsn = 19790477
```

### 1.3 两种增量备份模式

**Mode A（当前 ops_db 采用）：** 每个增量都基于最近的全量
```
full → incr1 (基于 full)
     → incr2 (基于 full)
     → incr3 (基于 full)
```

**Mode B（链式）：** 每个增量基于上一个增量
```
full → incr1 (基于 full)
     → incr2 (基于 incr1)
     → incr3 (基于 incr2)
```
Mode B 更节省空间，但 ops_db 当前**未实现**，只能手动用 xtrabackup 做。

## 二、备份模块

### 2.1 备份类型

| 类型 | 说明 | 备份工具 |
|------|------|---------|
| `full` | 全量备份 | xtrabackup |
| `incr` | 增量备份（基于最近全量） | xtrabackup |
| `dump` | 逻辑备份（SQL导出） | mysqldump |

### 2.2 本地全量备份

```bash
# 基本用法（本地 socket 连接）
python3 -m ops_db --yes backup --type full \
    --socket /var/run/mysqld/mysqld.sock \
    --user root \
    --password 'Aa123456%'

# 指定备份目录
python3 -m ops_db --yes backup --type full \
    --dest /data/backup \
    --user root --password 'Aa123456%'

# 备份并压缩
python3 -m ops_db --yes backup --type full \
    --compress \
    --user root --password 'Aa123456%'
```

**⚠️ 重要：--yes 必须放在 subcommand 前面**
```
正确：ops_db --yes backup ...
错误：ops_db backup --yes ...（不会跳过确认！）
```

**输出示例：**
```
✅ 全量备份完成: /data/backup/full_20260501_210000
   类型: full
   备份前 binlog: binlog.000025 @ position 157
   数据大小: 0.11 GB
```

### 2.3 增量备份

增量备份**只备份变化的数据**，比全量快很多，适合每天频繁执行。

```bash
# Step 1: 先做全量备份（这是增量链的起点）
python3 -m ops_db --yes backup --type full \
    --user root --password 'Aa123456%'

# Step 2: 增量备份 1（基于刚才的全量）
python3 -m ops_db --yes backup --type incr \
    --user root --password 'Aa123456%'

# Step 3: 增量备份 2（还是基于同一个全量，Mode A）
python3 -m ops_db --yes backup --type incr \
    --user root --password 'Aa123456%'

# Step 4: 增量备份 3
python3 -m ops_db --yes backup --type incr \
    --user root --password 'Aa123456%'
```

**输出示例：**
```
✅ 增量备份完成: /data/backup/incr_20260501_220000
   基于: /data/backup/full_20260501_210000
   LSN: 19790477 → 19790866
```

**验证增量备份的 LSN 链：**
```bash
# 查看全量备份的结束 LSN
cat /data/backup/full_20260501_210000/xtrabackup_checkpoints
# to_lsn = 19790477

# 查看增量1的开始 LSN，应该等于全量的 to_lsn
cat /data/backup/incr_20260501_220000/xtrabackup_checkpoints
# from_lsn = 19790477, to_lsn = 19790866 ✓ 链上了

# 查看增量2的开始 LSN，应该等于增量1的 to_lsn
cat /data/backup/incr_20260501_230000/xtrabackup_checkpoints
# from_lsn = 19790866, to_lsn = 19792556 ✓ 链上了
```

### 2.4 备份文件结构

```
/data/backup/
├── full_20260501_210000/          # 全量备份
│   ├── backup-my.cnf              # 备份时的 my.cnf 配置
│   ├── ib_buffer_pool             # InnoDB buffer pool 页
│   ├── ibdata1                    # InnoDB 系统表空间
│   ├── xtrabackup_binlog_info     # ← binlog 位置记录（用于主从）
│   ├── xtrabackup_checkpoints     # ← LSN 边界（核心！）
│   ├── xtrabackup_info            # 备份详细信息
│   └── mydb/                      # 数据库目录
│       ├── t1.ibd                 # 表数据文件
│       └── t2.ibd
├── incr_20260501_220000/          # 增量备份 1
│   ├── ib_buffer_pool
│   ├── xtrabackup_binlog_info
│   ├── xtrabackup_checkpoints     # ← LSN: from=19790477 → to=19790866
│   └── mydb/
│       └── t1.ibd                 # 只包含变化的部分
├── incr_20260501_230000/          # 增量备份 2
│   └── ...
└── backup_meta.json               # 元数据（备份时间、大小等）
```

### 2.5 备份过期清理

`--expire-days N`：备份后自动清理超过 N 天的**全量备份**。

```bash
# 保留 7 天
python3 -m ops_db --yes backup --type full \
    --expire-days 7 \
    --user root --password 'Aa123456%'
```

**注意：**
- 只清理 `full_*` 目录
- `incr_*` 增量备份**不会被自动清理**，需手动管理
- 如果删除了全量备份，对应的增量备份也失效了

### 2.6 远程备份（SSH）

```bash
python3 -m ops_db --yes backup --type full \
    --ssh-host 192.168.56.7 \
    --ssh-user root \
    --ssh-key ~/.ssh/id_rsa \
    --user root \
    --password 'Aa123456%'
```

## 三、恢复模块

### 3.1 恢复类型

| 类型 | 说明 | 场景 |
|------|------|------|
| `full` | 全量恢复 | 整体数据丢失 |
| `partial` | 单库恢复 | 只恢复特定数据库 |
| `pitr` | 时间点恢复 | 误删除数据，按时间点恢复 |
| `binlog-replay` | binlog 重放 | 精确到事务级别恢复 |

### 3.2 全量恢复

将整个数据库恢复到某个全量备份点。

```bash
python3 -m ops_db --yes restore --type full \
    --backup-dir /data/backup/full_20260501_210000 \
    --password 'Aa123456%' \
    --datadir /var/lib/mysql
```

**执行流程（ops_db 自动完成）：**
1. 停止 MySQL
2. 备份当前 datadir（自动 move 到 `xxx_old`）
3. 清空 datadir
4. xtrabackup --copy-back
5. chown mysql:mysql
6. 启动 MySQL

### 3.3 单库恢复（partial）

只恢复指定的一个或多个数据库，其他库保持不变。**这是最常用的恢复场景**，比如只想恢复某个误删的库。

```bash
python3 -m ops_db --yes restore --type partial \
    --backup-dir /data/backup/full_20260501_210000 \
    --databases mydb \
    --password 'Aa123456%'
```

**执行流程：**
1. 从备份中读取 mydb 的表定义
2. 在目标 MySQL 删除 mydb（如果存在）
3. 重建 mydb 数据库和所有表
4. 对每张表执行：
   ```
   ALTER TABLE t1 DISCARD TABLESPACE;    -- 丢弃当前表空间
   cp t1.ibd → /var/lib/mysql/mydb/     -- 复制备份的 ibd
   chown mysql:mysql t1.ibd              -- 修改权限
   ALTER TABLE t1 IMPORT TABLESPACE;     -- 导入表空间
   ```

**输出示例：**
```
✅ 2 张表已恢复
```

**注意：** 恢复时如果目标库已有同名数据库，会被**删除重建**。

### 3.4 PITR 时间点恢复（增量链恢复）

使用全量 + 多个增量，将数据恢复到某个时间点。这是增量备份的核心价值。

#### 3.4.1 为什么需要 PITR

假设：
- 周一做了全量备份
- 周二、周三、周四各做了一次增量备份
- 周四上午 10 点误删了数据

→ **PITR 可以恢复到 10 点之前的数据**，而不只是周一的备份点

#### 3.4.2 PITR 完整流程

**场景：** 全量 + 2 个增量，要恢复到第 2 个增量的时间点

```bash
# 假设备份结构：
# /data/backup/full_20260501_210000      (周一备份)
# /data/backup/incr_20260502_210000      (周二增量)
# /data/backup/incr_20260503_210000      (周三增量)

BACKUP_DIR=/data/backup/full_20260501_210000

# Step 1: 准备全量备份（加 --apply-log-only，只做日志应用，不做数据回放）
xtrabackup --prepare --apply-log-only \
    $BACKUP_DIR

# Step 2: 应用第一个增量（周二）
xtrabackup --prepare --apply-log-only \
    --incremental-dir=/data/backup/incr_20260502_210000 \
    $BACKUP_DIR

# Step 3: 应用第二个增量（周三）← 最后一次增量加 --apply-log-only
xtrabackup --prepare --apply-log-only \
    --incremental-dir=/data/backup/incr_20260503_210000 \
    $BACKUP_DIR

# Step 4: 最终 prepare（不加 --apply-log-only，这步才是真正回放数据）
xtrabackup --prepare $BACKUP_DIR

# Step 5: 复制回数据目录
xtrabackup --copy-back $BACKUP_DIR
chown -R mysql:mysql /var/lib/mysql
systemctl start mysql
```

**关键点解释：**

| 步骤 | 为什么要这样做 |
|------|--------------|
| `--apply-log-only` | 告诉 xtrabackup "只应用日志，不回放未提交的事务"，让数据保持一致状态 |
| 从旧到新应用增量 | LSN 链必须连续，不能跳过中间的增量 |
| 最后一步不加 `--apply-log-only` | 最后一次必须真正回放所有事务，让数据可读 |

**⚠️ 最重要的一条：copy-back 必须在所有增量 apply 完成后才能执行！**

如果在应用所有增量之前就 copy-back，LSN 链会断裂，数据会损坏。

#### 3.4.3 PITR 恢复的 LSN 验证

```bash
# Step 1 后：检查全量 prepare 结果
cat /data/backup/full_20260501_210000/xtrabackup_checkpoints
# to_lsn = 19790477

# Step 2 后：检查增量1应用后的 LSN
cat /data/backup/full_20260501_210000/xtrabackup_checkpoints
# last_lsn = 19790866  ← 应该是增量1的 to_lsn

# Step 3 后：检查增量2应用后的 LSN
cat /data/backup/full_20260501_210000/xtrabackup_checkpoints
# last_lsn = 19792556  ← 应该是增量2的 to_lsn
```

### 案例 2：单库误删恢复

**场景：** 操作失误删除了 `mydb` 库，其他库正常运行

```bash
# 查看有哪些备份
ls /data/backup/
```

### 案例 3：手动单表级恢复（高级）

当只想恢复某一张表，而不是整个库时，可以用 xtrabackup 的 `--export` 模式：

```bash
# 1. 在备份上做 export prepare
xtrabackup --prepare --export \
    /data/backup/full_20260501_020000

# 2. 在目标 MySQL 上执行（需要知道表结构）
mysql -u root -p -e "ALTER TABLE mydb.t1 DISCARD TABLESPACE;"

# 3. 复制 ibd 和 cfg 文件
cp /data/backup/full_20260501_020000/mydb/t1.ibd /var/lib/mysql/mydb/
cp /data/backup/full_20260501_020000/mydb/t1.cfg /var/lib/mysql/mydb/
chown mysql:mysql /var/lib/mysql/mydb/t1.ibd
chown mysql:mysql /var/lib/mysql/mydb/t1.cfg

# 4. 导入表空间
mysql -u root -p -e "ALTER TABLE mydb.t1 IMPORT TABLESPACE;"
```

## 六、备份恢复最佳实践

### 6.1 备份策略建议

```
每周一凌晨 2:00  全量备份  (full)
每天凌晨  2:00   增量备份  (incr)  基于最近的全量
保留             14-30 天  (--expire-days 14)
```

### 6.2 恢复基本原则

1. **恢复前先确认备份完整性**：检查 `xtrabackup_checkpoints` 文件，确认 LSN 链连续
2. **全量恢复会覆盖整个 datadir**：当前数据会被 move 到 `xxx_old`
3. **PITR 必须按时间顺序应用增量**：LSN 链不能断，不能跳过中间某个增量
4. **copy-back 必须在所有增量 apply 完成后**：这是最容易犯的错误
5. **partial 恢复只影响指定数据库**：其他库保持不变

### 6.3 备份前检查清单

```bash
# 1. 确认 MySQL 在运行
systemctl status mysql

# 2. 确认备份目录有足够空间（建议数据大小的 2-3 倍）
df -h /data/backup

# 3. 确认 xtrabackup 已安装
xtrabackup --version
```

## 七、常见问题

**Q: 增量备份可以跨全量吗？**
A: 当前工具实现（Mode A）要求所有增量都基于**同一个全量**。换全量后，增量链需要重新开始。

**Q: 密码有特殊字符（如 %）怎么办？**
A: 用单引号包裹即可，如 `--password 'Aa123456%'`，不需要转义。

**Q: 备份时 MySQL 需要停机吗？**
A: 不需要。xtrabackup 支持热备，MySQL 可以正常服务。建议开启 `innodb_flush_log_at_trx_commit=2`。

**Q: 远程备份和本地备份命令一样吗？**
A: 一样，只需加 SSH 参数 `--ssh-host`/`--ssh-key`。

**Q: 恢复时原数据会丢失吗？**
A: 全量恢复会自动将原 datadir move 到 `xxx_old` 备份，不会直接删除。

**Q: 增量备份恢复时 LSN 链断了怎么办？**
A: 必须从断点之前的备份重新开始。可以用 `xtrabackup --prepare` 检查每一步的 LSN 是否连续。

## 八、定时备份

ops_db 本身不提供内置调度，需配合系统 cron。

```bash
# 编辑 crontab
crontab -e

# 每周一凌晨 2 点全量备份
0 2 * * 1 cd /path/to/ops_db && python3 -m ops_db --yes backup --type full --password 'Aa123456%' --expire-days 14 >> /var/log/ops_db_backup.log 2>&1

# 每天凌晨 2 点增量备份（周二至周日）
0 2 * * 2-6 cd /path/to/ops_db && python3 -m ops_db --yes backup --type incr --password 'Aa123456%' >> /var/log/ops_db_backup.log 2>&1
```
